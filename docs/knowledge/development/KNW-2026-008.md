# KNW-2026-008: StrEnum voor type-safe constanten in plaats van losse string sets

## Metadata
- **Categorie:** development
- **Relevantie:** midden
- **Datum:** 2026-02-25
- **Sessie-context:** Review finding CON-001 — task statussen als plain strings i.p.v. StrEnum

## Inzicht

Wanneer een Python codebase al `StrEnum` gebruikt voor één set constanten (bijv. `SessionStatus`), moeten vergelijkbare sets (bijv. task statussen) ook als StrEnum gedefinieerd worden. Een losse `set` van strings mist IDE-autocompletion, is niet importeerbaar als type, en is fragiel bij refactoring. `StrEnum` is backward-compatible met strings maar voegt type safety toe.

## Context

`SessionStatus` was al een `StrEnum`, maar task statussen werden bijgehouden als:
```python
VALID_TASK_STATUSES = {"pending", "in_progress", "completed", "skipped"}
```

Dit werd door de convention-checker en python-specialist agents onafhankelijk gedetecteerd als inconsistentie.

## Geleerd

- `StrEnum` vergelijkingen werken transparant met plain strings (`TaskStatus.PENDING == "pending"` is `True`)
- `set(TaskStatus)` vervangt een handmatige string set voor validatie
- Na migratie naar StrEnum moeten alle plekken die hardcoded strings gebruiken (bijv. `_task_summary`) ook bijgewerkt worden — anders ontstaat functioneel correcte maar inconsistente code

## Toepassing

Bij elk Python 3.11+ project: als er één StrEnum bestaat, promoveer alle vergelijkbare string-constanten naar StrEnum. Dit voorkomt de "half-gemigreerd" toestand die AI reviewers terecht als inconsistentie flaggen.
